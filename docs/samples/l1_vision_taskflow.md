# TaskFlow ビジョン・要求ドキュメント

## 1. ドキュメントの立ち位置

### 1.1 目的

本書は、TaskFlow のビジョン・背景・要求を記録する。

### 1.2 範囲

- 対象ドメイン：個人・小規模チーム向けタスク管理
- 対象プロダクト：TaskFlow（Webアプリケーション）

### 1.3 粒度

- ソースコードは記載しない
- 技術詳細は L2（機能設計・技術方針）に委ねる
- 本書は「何のために・誰のために・どう使われるべきか」を表現する

---

## 2. プロダクトの目的・背景

### 2.1 解決したい課題・ペイン

- [x] 課題1：既存ツールは機能が多すぎて学習コストが高い
- [x] 課題2：複数デバイスでのタスク同期が面倒
- [x] 課題3：チームでの簡単なタスク共有ができない

### 2.2 想定ユーザー／ペルソナ

| ペルソナ | 特徴 | 主な利用シーン |
|---------|------|---------------|
| 田中さん（個人利用） | 30代フリーランス、複数案件を並行管理 | 毎朝のタスク確認、案件ごとの進捗管理 |
| 鈴木さん（チームリーダー） | 40代マネージャー、5人チームを管理 | メンバーへのタスク割り当て、進捗確認 |

### 2.3 ビジネス上のゴール

- [x] ゴール1：ローンチ3ヶ月でMAU 1,000人達成
- [x] ゴール2：有料プラン転換率 5%以上

### 5.2 利用シナリオ

#### シナリオ1：毎朝のタスク確認

**アクター**: 田中さん（個人利用）

1. 朝、PCでTaskFlowにログインする
2. 今日期限のタスク一覧を確認する
3. 優先度の高いタスクから着手する
4. 完了したタスクにチェックを入れる

**期待される結果**: 今日やるべきことが一目で把握でき、効率的に1日をスタートできる

#### シナリオ2：チームへのタスク割り当て

**アクター**: 鈴木さん（チームリーダー）

1. プロジェクト「新規サイト構築」を作成する
2. 必要なタスクを洗い出して登録する
3. 各タスクを担当メンバーに割り当てる
4. 週次でプロジェクトの進捗状況を確認する

**期待される結果**: チーム全体の作業状況が可視化され、遅延リスクを早期に発見できる

## 6. 非機能要求（高レベル）

### 6.1 性能

- ページ読み込み: 初回3秒以内、以降1秒以内
- タスク操作（登録・編集・完了）: 500ms以内に画面反映

### 6.2 セキュリティ

- パスワードはbcryptでハッシュ化して保存
- 通信はHTTPS必須
- セッションは24時間で自動タイムアウト

### 6.3 可用性

- サービス稼働率: 99.5%以上（月次）
- メンテナンス時間: 月2時間以内（深夜帯）

### 6.4 拡張性

- 初期: 同時接続1,000ユーザーを想定
- 将来: 10,000ユーザーまでスケール可能な設計

### 6.5 運用性

- アプリケーションログは30日間保持
- エラー発生時はSlack通知で即時検知

### 6.6 アクセシビリティ

- WCAG 2.1 Level AA準拠
- キーボードのみでの操作対応
- スクリーンリーダー対応（aria属性）

### 6.7 国際化・多言語対応

- 初期リリース: 日本語のみ
- 将来対応: 英語（UI国際化基盤は初期から準備）

### 6.8 データ保持・バックアップ

- ユーザーデータ保持期間: アカウント削除後30日
- データベースバックアップ: 日次（7日分保持）

### 6.9 監査・コンプライアンス

- ログイン履歴の保持（90日間）
- 個人情報アクセスログの記録

### 6.10 ライセンス・知的財産

- OSSライセンス: MIT、Apache 2.0、BSD系のみ使用可
- GPL系ライセンスは使用禁止
