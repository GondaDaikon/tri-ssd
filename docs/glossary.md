# Tri-SSD 用語集

Tri-SSD フレームワークで使用される用語の定義です。

## 基本概念

| 用語 | 定義 |
|------|------|
| **Tri-SSD** | Tri-Layer Slice Spec Driven。LLM/AI コードエージェントを前提とした仕様駆動開発フレームワーク |
| **三層モデル** | L1（ビジョン・要求）、L2（機能設計・技術方針）、L3（機能ドキュメント）の階層構造 |
| **分割統治** | 複雑な要件を機能単位に分割し、AI が扱える粒度に統治するアプローチ |
| **正本** | 各レイヤで管理される信頼できる唯一の情報源 |
| **NFR** | Non-Functional Requirement（非機能要求）。ID接頭辞は短縮形の NF を使用 |

## レイヤ

| 用語 | 定義 |
|------|------|
| **L1** | ビジョン・要求層。プロダクトの目的、ペルソナ、機能要求（REQ）、非機能要求を記述 |
| **L2** | 機能設計・技術方針層。技術選定、機能分解（F）、フェーズ設計（PH）、NFRカタログを記述 |
| **L3** | 機能ドキュメント層。機能単位の仕様、受け入れ条件、設計意図を記述 |
| **meta** | メタデータ層。テンプレートやガイドなど、Tri-SSD フレームワーク自体の情報 |

## ID 種別

| 用語 | 形式 | 定義 |
|------|------|------|
| **VISION** | VISION-YYYYMMDD-nnn | ビジョンドキュメント。L1 の主文書を識別 |
| **REQ** | REQ-YYYYMMDD-nnn | 機能要求。L1 で定義されるユーザー視点の要件 |
| **F** | F-YYYYMMDD-nnn | 機能。L2 で REQ を実装可能な単位に分解したもの |
| **PH** | PH-YYYYMMDD-nnn | フェーズ。「結合して動かして意味がある」単位で F を束ねたもの |
| **NF** | NF-YYYYMMDD-nnn | 非機能要求。L2 で具体化された性能・セキュリティ等の要件（※ NFR と同義、ID接頭辞として使用） |
| **SP** | SP-YYYYMMDD-nnn | スパイク。技術検証のための調査タスク |
| **RULES** | RULES-YYYYMMDD-nnn | 実装ルール。L2 で定義されるコード生成時の制約 |

## ドキュメント種別（kind）

| 値 | 定義 |
|----|------|
| **vision** | ビジョンドキュメント。L1 の主文書 |
| **foundation** | 技術基盤ドキュメント。L2 の主文書（用語集・技術方針・アーキテクチャ・NFR） |
| **phase** | フェーズドキュメント。L2 のフェーズ定義・機能一覧 |
| **rules** | 実装ルールドキュメント。L2 のコード生成制約 |
| **feature** | 機能ドキュメント。L3 の主文書 |

> **Note**: 要件（REQ）はL1ビジョン内でインライン定義、非機能要求（NF）はL2技術基盤内で定義します。

### kind と ID接頭辞の対応

| kind | ID接頭辞 | レイヤ |
|------|----------|--------|
| vision | VISION-YYYYMMDD-nnn | L1 |
| foundation | L2-YYYYMMDD-nnn | L2 |
| phase | PH-YYYYMMDD-nnn | L2 |
| rules | RULES-YYYYMMDD-nnn | L2 |
| feature | F-YYYYMMDD-nnn | L3 |

## ステータス

| 用語 | 定義 |
|------|------|
| **status** | ライフサイクル状態。active（有効）、deprecated（非推奨）、removed（削除済み） |
| **doc_status** | 文書・開発状態。draft（下書き）、reviewed（レビュー済み）、implemented（実装済み） |

## フロントマター

> 詳細な仕様は [frontmatter_spec.md](frontmatter_spec.md) を参照。

| 用語 | 定義 |
|------|------|
| **フロントマター** | Markdown ファイル冒頭の YAML メタデータブロック。`---` で囲まれる |
| **req_ids** | 機能が対応する要件 ID のリスト |
| **phase** | 機能が属するフェーズ ID |
| **nfr_ids** | 関連する非機能要求 ID のリスト |

## コマンド

| 用語 | 定義 |
|------|------|
| **/init-tri-ssd** | Tri-SSD ディレクトリ構造を初期化 |
| **/draft-l1** | L1 ドキュメントを作成（引数なし: 対話、引数あり: 変換） |
| **/draft-l2** | L1 から L2 技術基盤（foundation.md）を生成 |
| **/gen-phases** | L2 技術基盤からフェーズ定義・機能一覧を生成 |
| **/draft-rules** | L2 実装ルールのたたき台を生成 |
| **/gen-l3** | L2 から L3 ドキュメントを生成 |
| **/gen-code** | L3 からコード・テストを生成 |
| **/check** | ドキュメント整合性をチェック |
| **/review** | AI によるドキュメントレビュー + ステータス昇格 |

## その他

| 用語 | 定義 |
|------|------|
| **トレーサビリティ** | 要件から実装までの追跡可能性。REQ → F → コードの連鎖 |
| **Exit Criteria** | フェーズ完了の判定基準 |
| **受け入れ条件** | 機能が完了したとみなす条件。Given-When-Then 形式を推奨 |
| **軽量運用** | 小規模・単純な機能向けの簡略化されたL3運用 |
| **実装ルール** | AI/人間がコード実装時に守るべきルール。L2で定義し、L3実装のフィードバックで進化させる |
