# SSDD 用語集

SSDD フレームワークで使用される用語の定義です。

## 基本概念

| 用語 | 定義 |
|------|------|
| **SSDD** | Slices Specification-Driven Development。LLM/AI コードエージェントを前提とした仕様駆動開発フレームワーク |
| **三層モデル** | L1（ビジョン・要求）、L2（機能設計・技術方針）、L3（機能ドキュメント）の階層構造 |
| **分割統治** | 複雑な要件を機能単位に分割し、AI が扱える粒度に統治するアプローチ |
| **正本** | 各レイヤで管理される信頼できる唯一の情報源 |

## レイヤ

| 用語 | 定義 |
|------|------|
| **L1** | ビジョン・要求層。プロダクトの目的、ペルソナ、機能要求（REQ）、非機能要求を記述 |
| **L2** | 機能設計・技術方針層。技術選定、機能分解（F）、フェーズ設計（PH）、NFRカタログを記述 |
| **L3** | 機能ドキュメント層。機能単位の仕様、受け入れ条件、設計意図を記述 |
| **meta** | メタデータ層。テンプレートやガイドなど、SSDD フレームワーク自体の情報 |

## ID 種別

| 用語 | 形式 | 定義 |
|------|------|------|
| **REQ** | REQ-YYYYMMDD-nnn | 機能要求。L1 で定義されるユーザー視点の要件 |
| **F** | F-YYYYMMDD-nnn | 機能。L2 で REQ を実装可能な単位に分解したもの |
| **PH** | PH-YYYYMMDD-nnn | フェーズ。「結合して動かして意味がある」単位で F を束ねたもの |
| **NF** | NF-YYYYMMDD-nnn | 非機能要求。L2 で具体化された性能・セキュリティ等の要件 |
| **SP** | SP-YYYYMMDD-nnn | スパイク。技術検証のための調査タスク |

## ドキュメント種別（kind）

| 値 | 定義 |
|----|------|
| **vision** | ビジョンドキュメント。L1 の主文書 |
| **req** | 要件ドキュメント。個別の REQ を詳細に記述 |
| **feature** | 機能ドキュメント。L3 の主文書 |
| **nfr** | 非機能要求ドキュメント |
| **phase** | フェーズドキュメント |
| **spike** | スパイクドキュメント |
| **overview** | 概要ドキュメント。L2 の主文書 |

## ステータス

| 用語 | 定義 |
|------|------|
| **status** | ライフサイクル状態。active（有効）、deprecated（非推奨）、removed（削除済み） |
| **doc_status** | 文書・開発状態。draft（下書き）、reviewed（レビュー済み）、implemented（実装済み） |

## フロントマター

| 用語 | 定義 |
|------|------|
| **フロントマター** | Markdown ファイル冒頭の YAML メタデータブロック。`---` で囲まれる |
| **req_ids** | 機能が対応する要件 ID のリスト。L3 必須フィールド |
| **phase** | 機能が属するフェーズ ID。L3 必須フィールド |
| **nfr_ids** | 関連する非機能要求 ID のリスト |

## コマンド

| 用語 | 定義 |
|------|------|
| **/init-ssdd** | SSDD ディレクトリ構造を初期化 |
| **/draft-l1** | 対話形式で L1 ドキュメントを作成 |
| **/gen-l2** | L1 から L2 ドキュメントを生成 |
| **/gen-l3** | L2 から L3 ドキュメントを生成 |
| **/check** | ドキュメント整合性をチェック |
| **/review** | AI によるドキュメントレビュー |
| **/promote-status** | doc_status を次の段階に昇格 |
| **/propagate-change** | 変更影響を分析し関連ドキュメントへの伝播を支援 |

## その他

| 用語 | 定義 |
|------|------|
| **トレーサビリティ** | 要件から実装までの追跡可能性。REQ → F → コードの連鎖 |
| **Exit Criteria** | フェーズ完了の判定基準 |
| **受け入れ条件** | 機能が完了したとみなす条件。Given-When-Then 形式を推奨 |
| **軽量運用** | 小規模・単純な機能向けの簡略化されたL3運用 |
